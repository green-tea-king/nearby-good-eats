## 1. 專案一句話

手機使用者在現場,用所在位置（或選地區）找出**步行可達、評價最好的前五家餐廳**,結合 Google 評分／評論數與台灣餐廳評鑑（米其林等）做排名,並用 AI 解讀「為什麼上榜」。

```

  ↓
```

## 4. 使用的 Google API

- **Maps JavaScript API** — 顯示地圖、標記
- **Places API (New)** — 搜尋餐廳、取得評分／評論數等
- **Geolocation**（瀏覽器原生）— 取得使用者所在地

### Places API (New) 可用欄位（已查證）

| 用途 | 欄位 | 計費等級 |
|---|---|---|
| 評分 | `rating` | Enterprise |
| **評論數** | `userRatingCount` | Enterprise |
| 價位 | `priceLevel` / `priceRange` | Enterprise |
| 營業狀態 | `currentOpeningHours` | Enterprise |
| 官方簡介 | `editorialSummary` | Ent + Atmosphere |
| AI 評論摘要 | `reviewSummary` / `generativeSummary` | Ent + Atmosphere |
| 個別評論 | `reviews` | Ent + Atmosphere |

> 注意：`reviews`、各種 summary 屬較貴的 Atmosphere SKU。內部測試無妨,但別無腦全抓。

### Google **拿不到**的（別把邏輯建在這上面）
- ❌ 按讚數（Google Maps 無此概念）
- ❌ 米其林／餐廳評鑑等級（非 Google 資料,須自建名單,見 §6）

## 5. 排名機制

### 核心：貝氏加權平均

```
加權分數 = (C × m + userRatingCount × rating) / (C + userRatingCount)
```

- `rating`：該店評分
- `userRatingCount`：評論數（越多越可信）
- `m`：全體平均分（基準值）
- `C`：可信度門檻（評論數要多少才算「夠數」,可調）

> 效果：「5.0 分但僅 3 則評論」會被拉回接近平均,「4.6 分但 2000 則」才會浮上來 —— 反映真實口碑。

### 加分項（加在貝氏分數之上）

| 來源 | 加分 |
|---|---|
| 米其林三星 | +0.5 |
| 米其林二星 | +0.4 |
| 米其林一星 | +0.3 |
| 必比登推介 | +0.2 |
| 500 盤 | 每盤 +0.03（上限 +0.4） |
| 步行距離近 | 影響排名（權重待實作時定） |
| 目前營業中 | 影響排名／或作為篩選 |

> 所有權重集中在單一設定處,改一個數字即生效。

### 排序選項（使用者可切換）
- **綜合**（預設,上述完整公式）
- **純距離**
- **純評分**

## 6. 餐廳評鑑名單（後台可維護）

### 收錄三種評鑑（第一版先建**台北**）
- **米其林星**（三星／二星／一星）
- **必比登推介**（平價美食,最貼「走路覓食」）
- **500 盤**（台灣在地觀點,依盤數）

### 名單存放：獨立 JSON 檔（做法 1）
- 與程式分離,維護＝改 JSON + push,不動程式碼
- 將來可升級為 Google Sheet 後台（比對機制共用,不白做）

### 資料結構（支援多評鑑、多徽章）
```json
{
  "name": "餐廳名",
  "address": "輔助比對用地址",
  "awards": [
    { "guide": "michelin", "level": "一星", "year": 2025 },
    { "guide": "500plate", "level": "6盤", "year": 2025 },
    { "guide": "bib", "year": 2025 }
  ]
}
```

### 比對機制（注意難點）
- 用「**店名正規化 + 地址輔助**」做**模糊比對**,不能用完全相等
  - 處理全半形、分店名、中英文、「餐廳」二字有無等差異
- **500 盤特例**：原始以「菜」為單位投票,比對時須以**餐廳層級彙總**盤數
- 比對結果分「**高信心自動標**」與「**低信心待人工覆核**」,避免默默標錯
- 徽章顯示須帶**年份**（評鑑每年更新）

## 7. AI 角色（定位在「第 3 層：結果解讀」）

AI 只**解讀與表達**,不碰計算與定位（排名靠公式、距離靠程式,確保可重現精確）。資料來源一律是 Places 回傳的真實資料,不瞎掰。

### 第一版做（最有感）
1. **上榜理由** — 每家一句,說明為何排此名
2. **評論摘要** — 將評論濃縮成一句特徵

### 延後
3. **情境比較** — 五家整體短評（「想快選 A、想坐久選 B」）

### 實作要點
- **一次呼叫處理五家**（把五家資料一起丟給 Claude,回傳五段）→ 省錢又快,不要跑五次
- 透過前端呼叫 Claude API（Anthropic /v1/messages,artifact 環境免帶金鑰）
- 模型用 `claude-sonnet-4-6`

## 8. 使用者介面

### 風格基調
- **暗色系為主**、簡約質感、現代設計、**工具感**
- 易讀性優先：深炭灰底（非純黑）、近白文字（非純白）、對比達 WCAG AA
- 文字數字為主、照片為輔；配色克制；操作即時；動效收斂（150–200ms）

### 版面（手機優先）
- **開場**：大鈕「找我附近的餐廳」（自動定位）+ 小字「或選地區」
- **結果**：清單為主,卡片緊湊;浮動鈕切換整頁地圖
- **清單與地圖不並排**（手機螢幕不夠寬）

### 卡片資訊
**主卡片（一眼可見）**：排名①~⑤、店名、評分、評鑑徽章、評論數、價位、營業中、步行距離/時間、AI 一句理由、`[導航] [詳情▾]`

**展開詳情**：照片、評論摘要、營業時間、地址、電話

### 頂部控制
- 範圍（可點改,內含半徑滑桿）
- 「只看營業中」開關
- 排序切換（綜合／距離／評分）

### 設計 tokens（暗色,已用於預覽）
```
--bg:#101216        底（深炭灰,非純黑）
--surface:#181B20   卡片
--surface-2:#1F232A 展開／次層
--line:#262B33      細分隔線
--text:#E8EAED      主文字（近白）
--text-dim:#9BA3AE  次要文字
--accent:#E4B84C    主點綴（沈穩金）
--green:#3DD68C     營業中
徽章：米其林 #FF5A5F／必比登 #FF9F43／500盤 #5AC8FA（皆低彩度深底+亮字）
```
> 顏色集中為 CSS variables,將來要加淺色模式只需多定義一組值。

### 步行體驗設計
- 每家顯示「步行 X 分 · Y 公尺」
- 搜尋半徑取步行尺度（預設約 800m,可調）
- 「現在營業中」醒目標記

### 字體
- 中文：Noto Sans TC
- 數字／英文：Space Grotesk（tabular 對齊）

## 9. 重要決策紀錄（為什麼這樣選）

### 為何放棄「里」改用「區」
原本希望精確到「里」以支援步行,但評估後放棄,因為：
1. **湊不滿五家**：一個里可能只有 2~3 家甚至 0 家餐廳
2. **Places 不保證回傳里級地址**
3. **座標系統地雷**：村里界圖資是 TWD97,Google 是 WGS84,需轉換
4. **圖資工程重**：要下載 SHP、轉 GeoJSON、按區拆檔、point-in-polygon

→ 改用「區搜尋 + 步行半徑 + 顯示步行距離」即足以支撐「走路能找到」,且砍掉最重、最易錯的部分。對「走路找吃的」場景,**以定位點畫步行半徑甚至比選整個區更準**。

### 為何純前端 + GitHub Pages（而非後端）
- 內部測試、小流量,後端代理價值不大
- 純前端最快上線、最易維護
- 金鑰靠 referrer + API 範圍限制保護,測試足夠
- 將來真要對外大流量,再把 Places 呼叫搬到後端,前端 UI 大多可留用

### 為何用貝氏加權而非純評分
純評分會讓「5.0 分／3 則評論」的冷門小店灌上前五,失去參考價值。
